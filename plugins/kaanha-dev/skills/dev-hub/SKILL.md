# kaanha dev hub

All dev servers on this machine are managed from ONE registry, in the ops
home: `<KAANHA_HOME>/registry.json` (`KAANHA_HOME` if set, else
`~/.claude/kaanha`). Every project gets a unique assigned port so servers
run side by side without conflicts. The script is bundled in this plugin at
`${CLAUDE_PLUGIN_ROOT}/scripts/kaanha_dev.py` and its SessionStart hook runs
`scan` each session — which also **auto-covers every enrolled project in the
24/7 fleet** (see the kaanha-agents routines skill).

## Commands (run from anywhere)

Use the bundled script — substitute the plugin path (`${CLAUDE_PLUGIN_ROOT}`
resolves to this plugin's install dir):

```
python "<plugin>/scripts/kaanha_dev.py" list      # all projects + ports
python "<plugin>/scripts/kaanha_dev.py" status    # what's running
python "<plugin>/scripts/kaanha_dev.py" start <name> [<name>...]
python "<plugin>/scripts/kaanha_dev.py" stop <name|all>
python "<plugin>/scripts/kaanha_dev.py" logs <name> [-n 100]
python "<plugin>/scripts/kaanha_dev.py" sync      # regenerate launch.json files
python "<plugin>/scripts/kaanha_dev.py" scan      # auto-enrol new projects + sync fleet
```

First run auto-bootstraps the ops home (registry + fleet config). Set
`projectsRoot` in `registry.json` to where your repos live so `scan` can find
them. Servers start detached (they survive the session) and log to
`<KAANHA_HOME>/logs/<name>.log`. `stop` only stops servers the hub started.

## Rules for Claude

1. **Previewing the CURRENT project in the Browser pane**: prefer
   `preview_start` with the config name from this project's
   `.claude/launch.json` (generated by `sync` from the registry) - the
   pane manages that server's lifecycle itself.
2. **Running OTHER projects, or several at once** (e.g. the site plus an
   API it calls): use the hub `start` command with the registered names.
3. **Before starting anything**, run `status` - the server may already be
   up; never start a duplicate.
4. **Port conflicts**: ports are assigned in the registry - fix the
   registry (then `sync`), don't improvise a different port inline.
5. **New project**: usually automatic — `scan` enrols any project under
   `projectsRoot` with a `dev` script and covers it in the fleet. To add one
   by hand, put an entry in `registry.json` (unique port; use
   `env: {"PORT": ...}` for servers that read the PORT variable, or pass
   `-p`/`--port` args for Next/Vite) and run `sync`.
6. After starting a server, verify it with `status` (or probe the port)
   before telling the user it's up; check `logs <name>` when it isn't.
