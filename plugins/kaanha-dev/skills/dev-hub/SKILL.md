# dev hub

All dev servers on this machine are managed from ONE registry:
`registry.json`, kept next to `scripts/kaanha_dev.py` (copy
`scripts/registry.sample.json` there on first use and edit it). Every
project gets a unique assigned port so servers run side by side without
conflicts.

## Setup (once)

1. Copy `scripts/registry.sample.json` to `scripts/registry.json`; set
   `projectsRoot` to the folder that holds your repos, and list your
   projects (or leave the list empty and let `scan` fill it).
2. Windows only: copy `scripts/withnode.sample.cmd` to
   `scripts/withnode.cmd` and set your Node.js install directory inside.
   Claude sessions often lack your interactive shell's PATH; routing
   every server through this wrapper fixes pnpm/node resolution. Set each
   registry entry's `runtimeExecutable` to the wrapper's absolute path
   and put the real tool (`pnpm`/`npm`) as the first runtimeArg.

## Commands (run from anywhere)

```
python <path-to>/kaanha_dev.py list      # all projects + ports
python <path-to>/kaanha_dev.py status    # what's running
python <path-to>/kaanha_dev.py start <name> [<name>...]
python <path-to>/kaanha_dev.py stop <name|all>
python <path-to>/kaanha_dev.py logs <name> [-n 100]
python <path-to>/kaanha_dev.py sync      # regenerate per-project launch.json
python <path-to>/kaanha_dev.py scan      # auto-enroll new projects
```

Servers start detached (they survive the session) and log to
`logs/<name>.log` next to the script. `stop` only stops servers the hub
started. `scan` registers any project under `projectsRoot` that has a
package.json dev script, assigns the next free port, and respects the
registry's `ignore` list.

## Rules for Claude

1. **Previewing the CURRENT project in the Browser pane**: prefer
   `preview_start` with the config from this project's
   `.claude/launch.json` (generated by `sync`).
2. **Running OTHER projects, or several at once**: use the hub `start`
   command with the registered names.
3. **Before starting anything**, run `status` - never start a duplicate.
4. **Port conflicts**: fix the registry (then `sync`), don't improvise a
   different port inline.
5. After starting a server, verify with `status` or a port probe before
   reporting it up; check `logs <name>` when it isn't.
